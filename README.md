# hashharness

## Goal

A small, append-only text store that gives you a **trustworthy historical record**: once something is written, the system can prove what was said, when it was accepted, against which schema, and how it links to anything else — all without trusting the calling client.

The whole design is organized around one question: *if a reader looks at a chain of records six months from now, what can they actually prove from cryptographic evidence alone, versus what are they taking on faith?* Every feature in this repo exists to push more things into the "provable" column.

## Use cases

- **Append-only investigations / debugging logs.** Hypotheses, evidence, model changes, decision records — anything where the *order in which beliefs evolved* matters and "who edited what later" must be impossible.
- **Workflow / task chains** where each state transition must descend from the previous one. The chain head pointer enforces that two workers can't both claim "I'm the next step after X."
- **Schema-evolving structured datasets** where you need to validate old records against the rules in force *when they were written*, not the current rules. Schemas themselves are append-only and hash-chained.
- **MCP-served knowledge bases** for LLM agents that need to reason about a body of evidence and prove they didn't fabricate citations.
- **Compaction / summarization workflows** that must preserve verifiable provenance back to source records.

## Trust invariants

These are what `verify_chain` checks. If verification passes, all of them hold for every record in the walk:

- **Content immutability.** The storage key is `sha256(text)` (`text_sha256`); rewriting the same text with different metadata or links is rejected.
- **Record integrity.** Every record carries `meta_sha256`, `links_sha256`, and `record_sha256 = sha256(text_sha256 + meta_sha256 + links_sha256)`. Tampering with any field that's hashed is detected on read.
- **Server-stamped time.** `created_at` is generated by the server at write time and folded into `meta_sha256`. Callers cannot supply or backdate it. The MCP layer explicitly rejects caller-supplied `created_at`.
- **Links pin record identity.** Inter-record links carry `record_sha256` (full record hash), not just `text_sha256`. A link therefore commits to the target's exact text *and* metadata *and* links — not just its text content.
- **No silent forks on chains.** Link rules can be marked `chain_predecessor: true`. For chain types, the server tracks a per-`(work_package_id, item_type)` head pointer and uses compare-and-swap on `create_item`: a new record's predecessor link must equal the current head, otherwise the write is rejected with `"head moved"`. Concurrent forks cannot both succeed.
- **Schemas are append-only and hash-chained.** Each `set_schema` call adds a new immutable version that names its predecessor; `expected_prev` must equal the current schema head (compare-and-swap). Stale or concurrent schema updates are rejected with `"schema head moved"`.
- **Records are bound to the schema active at write time.** Every record carries `schema_sha256` (the schema head's hash at write time) and `schema_binding_sha256 = sha256({record_sha256, schema_sha256})`. `verify_chain` resolves each record's schema version and re-validates that record's links against *those* rules — not whatever the schema looks like today. A "loosen → write malicious record → tighten" attack leaves a permanent, auditable trace.
- **Cryptographic walk.** `verify_chain` recomputes every hash it can and reports per-item `ok`/`errors`. With `summary=true` it returns just `ok`, `checked_items`, `errors_count`, and `root_text_sha256`.

What's **not** invariant (open gaps): writes are unauthenticated (anyone reaching the protocol can write to any `work_package_id`), `set_schema` itself is unauthenticated, and there are no resource caps. See the security roadmap for ranked mitigations.

## Stored item shape

```json
{
  "type": "HypothesisChange",
  "text_sha256": "<sha256(text)>",
  "meta_sha256": "<sha256(metadata)>",
  "links_sha256": "<sha256(links)>",
  "record_sha256": "<sha256(text_sha256 + meta_sha256 + links_sha256)>",
  "schema_sha256": "<record_sha256 of the schema version active at write time>",
  "schema_binding_sha256": "<sha256({record_sha256, schema_sha256})>",
  "work_package_id": "wp-123",
  "created_at": "2026-04-25T10:30:00+00:00",
  "title": "Updated hypothesis",
  "attributes": { "author": "alice", "status": "draft" },
  "text": "....",
  "links": {
    "prevHypothesisChange": "<target record_sha256>",
    "evidences": ["<target record_sha256>", "<target record_sha256>"],
    "evidencesHash": "<sha256(sorted target record_sha256 values)>"
  }
}
```

`text_sha256` is the **storage key** (filename / sqlite PK; used to fetch by content). `record_sha256` is the **link id** (used by other records to refer to this one). They diverge whenever metadata is non-empty.

## Schema

User-defined and typed. Each entry under `types` declares an item type and the link fields it may use.

```json
{
  "types": {
    "Evidence": { "links": {} },
    "HypothesisChange": {
      "links": {
        "prevHypothesisChange": {
          "kind": "single",
          "target_types": ["HypothesisChange"],
          "chain_predecessor": true
        },
        "evidences": {
          "kind": "many",
          "target_types": ["Evidence"]
        }
      }
    }
  }
}
```

For every `many` link field, the server stores a derived `<field>Hash` value over the sorted list of referenced `record_sha256` values (`evidencesHash` for `evidences`).

### `chain_predecessor`

Opt-in flag on a `kind: "single"` link rule. Only one predecessor link per type is allowed. When set, the server tracks a per-`(work_package_id, item_type)` head; the first item must omit the predecessor link, every subsequent item must supply `record_sha256` of the current head, and the head advances atomically on success. `find_tip` returns the head record directly (O(1)) instead of scanning by `created_at`. Without this flag, link rules are ordinary single-target references and offer no fork protection.

### Schema versioning

`set_schema(schema, expected_prev)` performs compare-and-swap on the schema chain head. Pass `expected_prev: null` (or omit) for the first/genesis schema; otherwise pass the current head's `record_sha256`. Each schema version is stored as:

```json
{
  "record_sha256": "<sha256({prev_schema_sha256, payload_sha256, created_at})>",
  "prev_schema_sha256": "<previous head, or null for genesis>",
  "payload_sha256": "<sha256(payload)>",
  "created_at": "<server-stamped>",
  "payload": { "types": { ... } }
}
```

Schema versions live under `data/schemas/<record_sha256>.json` (filesystem) or the `schema_versions` table (sqlite). The current head is in `data/schemas/HEAD` or the `schema_head` row.

Pre-versioning data is migrated lazily on store init: the previously stored single schema becomes the genesis version, every existing item is stamped with `schema_sha256` + `schema_binding_sha256`, and `record_sha256` values do not change so links remain valid. For full link-format migration (text_sha256 → record_sha256) on legacy data, see the one-shot tool in `src/hashharness/migrate_legacy.py` (untracked utility script).

## MCP tools

| Tool | Purpose |
|---|---|
| `set_schema` | Append a new schema version. CAS via `expected_prev`. |
| `get_schema` | Return the current schema payload, or a historical one via `at: <record_sha256>`. |
| `get_schema_history` | Full schema chain from genesis to current head. |
| `get_schema_version` | One schema version (full record) by `record_sha256`. |
| `create_item` | Create an immutable record. `created_at` is server-stamped (caller-supplied is rejected). Link values are target `record_sha256` (not `text_sha256`). For chain types, the predecessor link must match the current head. Accepts `return: "minimal" \| "full"`; default `minimal` returns just `text_sha256` and `record_sha256`. |
| `find_items` | Substring (default) or regex (`regex=true`) search over `text`, `title`, `work_package_id`, or `all`. Optional `fields` projection and exact-match `attributes` filter. |
| `get_item_by_hash` | Fetch one record by `text_sha256`. |
| `get_work_package` | Return every record in one `work_package_id`, optionally filtered by `type`. |
| `find_tip` | Return the chain head for `(work_package_id, item_type)`. For chain types, O(1) head lookup; otherwise picks max-`created_at` (legacy fallback). |
| `query_chain` | Walk from a root `text_sha256`, following `record_sha256` links transitively; return all records in the chain. |
| `verify_chain` | Recompute every hash for every record reachable from a root `text_sha256`. Reports per-item `ok`/`errors`. With `summary=true`, returns only `ok`, `checked_items`, `errors_count`, `root_text_sha256`. |

## Storage backends

- **Filesystem** (`data/items/<text_sha256>.json` per record, `data/schemas/<record_sha256>.json` per schema version, `data/heads.json` and `data/schemas/HEAD` for head pointers). Plain JSON; directly grep-able.
- **Sqlite** (`items`, `schema_versions`, `schema_head`, `heads`, `schema_kv` tables). WAL journal mode.

Records are cached in memory by `work_package_id`. A cached work package stays resident for 5 minutes after its last use; any access refreshes the TTL for the whole package.

## Run

Start the MCP server over stdio:

```bash
PYTHONPATH=src python3 -m hashharness.mcp_server
```

Choose a storage directory or backend via env vars:

```bash
HASHHARNESS_STORAGE_BACKEND=sqlite \
HASHHARNESS_DATA_DIR=/path/to/hashharness.sqlite \
PYTHONPATH=src python3 -m hashharness.mcp_server
```

Over HTTP:

```bash
HASHHARNESS_MCP_TRANSPORT=http \
HASHHARNESS_HTTP_HOST=127.0.0.1 \
HASHHARNESS_HTTP_PORT=8000 \
PYTHONPATH=src python3 -m hashharness.mcp_server
```

HTTP transport exposes `POST /mcp` for JSON-RPC and `GET /health`. Notifications return `202 Accepted`; requests with an `id` return the JSON-RPC response body.
